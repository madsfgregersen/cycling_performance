Cycling Performance — Architecture Canvas

A private, personal cycling-training app for one athlete building toward the Geo Park Gran Fondo (30 Aug 2026). Python/FastAPI + Postgres on Railway, with two surfaces (a web dashboard and a Telegram bot) over a shared backend, and an LLM-backed coach layered on top. Read top→bottom: data flows in from the left, is stored in six buckets, computed into readiness, interpreted by the coach, and delivered to the two surfaces.

Key design decisions — the learning layer
Sacred separation
Actual history (bucket 5, computed-only) and planned/provisional data (bucket 6) never mix — in data or in the coach's prose. The coach explains the verdict; it never re-derives it.
Propose → confirm → write
Any coach action that changes the calendar proposes first, waits for an explicit yes, then writes. Guardrails (taper/event date) are enforced in code twice — at propose and at apply — never left to the prompt.
One coach brain, thin adapters
Dashboard and Telegram are two I/O adapters onto one orchestrator + one shared voice. The LLM is reached only through ai_coach, which fails safe and logs cost.
Compute-on-demand projection
The forward CTL/ATL/TSB curve is computed from planned workouts when asked — never written back to the sacred readiness bucket.
Event-driven fan-outs
Two webhooks + one authed POST drive everything; each proactive push dedupes on its own key, so re-arriving data the same day is a no-op.
